Super Agent 工具系统设计文档
=============================

一、工具注册机制（已更新）（已更新）
每个工具通过 ToolRegistry 统一注册，包含名称、描述、参数 Schema 和执行函数。
注册时自动校验工具名称唯一性，避免命名冲突。
支持批量注册：registry.register(...allTools)

二、执行管线
工具的执行不是直接调用 execute 函数那么简单。生产级 Agent 的工具执行是一条管线：
1. 参数校验：根据 JSON Schema 校验输入，拦截无效参数
2. 权限检查：确认当前用户/会话有权限执行该工具
3. 执行函数：调用 tool.execute(input)
4. 结果截断：防止超大结果撑爆上下文窗口
5. 结果格式化：统一输出格式，方便模型理解

三、结果截断策略
为什么需要截断？想象模型调用 read_file 读取一个 10 万行的日志文件——
如果把完整内容塞进上下文，一次调用就消耗了大量 token，挤压了后续推理空间。

截断策略采用 Head/Tail 60/40 分割：
- 保留前 60% 的内容（通常包含文件头部、重要声明）
- 保留后 40% 的内容（通常包含最新的日志、函数结尾）
- 中间部分用 "[省略 N 字符]" 标记

这比单纯的 head 截断更智能——很多时候文件尾部的信息比中间更有价值。

四、并发控制
并发安全性不是按工具名决定的，而是按具体输入判断：
- get_weather：只读操作，天然并发安全
- read_file：只读操作，并发安全
- write_file：有副作用，必须串行执行
- bash "ls"：只读命令，可以并发
- bash "rm -rf /"：危险操作，必须串行且需要权限确认

五、工具元数据
每个工具携带元数据，供 Agent Loop 做决策：
- isConcurrencySafe: boolean — 是否可以和其他工具并行执行
- isReadOnly: boolean — 是否只读（不修改系统状态）
- maxResultChars: number — 结果最大字符数
- description: string — 给模型看的工具描述

六、最佳实践
1. 工具描述要写清楚"什么时候用"和"什么时候不要用"
2. 参数要有明确的类型和默认值
3. 错误信息要对模型友好——模型需要理解为什么失败才能换策略
4. 结果格式要结构化——JSON 比自然语言更容易被模型准确解析
